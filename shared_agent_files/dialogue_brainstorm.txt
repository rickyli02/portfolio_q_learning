Topic: WRDS Real-Data Experiment Path
Relevance: 9.5/10.0
Difficulty: 7.0/10.0
Why it matters:
- this is the most direct way to test whether the current synthetic and theorem-aligned stack has practical value outside controlled GBM-style assumptions
- it also forces the repo to become explicit about data-timestamp contracts, observability, leakage prevention, and evaluation methodology

Candidate directions:
- define a first narrow real-data contract rather than a broad market-data platform
- recommended first design slice:
  - asset universe: start with liquid US equities only
  - frequency: daily first, then optionally weekly or monthly before any intraday work
  - target task: portfolio allocation over a fixed rebalance cadence
  - conditioning set: begin with price/return/volatility-style observable features only
  - benchmark layer: compare CTRL against simple classical baselines, not only the synthetic oracle
- define the anti-leakage contract explicitly:
  - what timestamp each raw field is observed
  - what transformations are allowed at decision time
  - how train/validation/test splits are cut in calendar time
  - how reconstitutions / delistings / survivorship bias are handled
- define evaluation axes explicitly:
  - rebalance frequency
  - transaction-free vs transaction-cost-aware evaluation
  - market regime segmentation
  - asset-subset robustness
- defer options/macros/multimarket inputs until the first equity-only contract is stable

Recommended next documentation target:
- `codex_files/repo_design_memory.md`

Related areas in code structure:
- future dataset-loading seam beyond current synthetic-only setup
- `src/features/` for observable real-data conditioning inputs
- `src/eval/` and `src/backtest/` for real-data experiment summaries

Agent opinion:
- the main design risk is trying to support too many markets and frequencies at once
- the right first question is not "can the repo ingest WRDS?" but "what is the smallest leakage-safe real-data experiment contract worth supporting?"

Topic: Alternative Synthetic Data Stress Ladder
Relevance: 9.0/10.0
Difficulty: 6.5/10.0
Why it matters:
- richer synthetic regimes are the cleanest way to study where CTRL keeps working, where it degrades, and why
- they preserve interpretability better than jumping directly into messy real-world data

Candidate directions:
- structure this as a staged stress ladder instead of a grab-bag:
  - stage 1: time-varying drift and volatility
  - stage 2: jump-diffusion
  - stage 3: stochastic volatility
  - stage 4: heavier-tail / GARCH-like / econophysics-inspired return processes
  - stage 5: non-Markov or memory-bearing variants
- keep the design question consistent across stages:
  - what theoretical assumption is being relaxed
  - what behavior should CTRL plausibly retain
  - what failure mode would be informative rather than merely "bad performance"
- sharpen the currently vague subideas:
  - "market surprise" should mean a concrete latent-regime jump, event shock, or parameter break
  - "insider trading" should become a specific asymmetric-information toy setup or be dropped
  - "pairs trading behavior" should specify whether it means cointegration, relative-value mean reversion, or spread-state conditioning
- compare all richer synthetic variants back to:
  - the baseline GBM environment
  - the same evaluation metrics
  - the same explanation template for what the model captures and misses

Recommended next documentation target:
- `shared_agent_files/claude_code_todo.md`

Related areas in code structure:
- `src/envs/` for new synthetic environment families
- future config/environment dispatch seams
- `src/eval/` for consistent regime-comparison reporting
