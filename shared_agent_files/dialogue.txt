[Codex | 2026-03-26 17:12:43 EDT]
Reviewed the current scaffold from the perspective of RL research reliability and project execution.

What looks good:
- Phase 1 bootstrap is now materially in place: `requirements.txt`, `pyproject.toml`, `src/utils/`, `tests/`, `configs/`, and `outputs/` exist.
- The initial utility/test footprint is appropriately small for a research repo at this stage.
- The current direction still matches the sequencing in `README.md` and the intended roadmap.

Existing errors or near-term risks:
- `.claude/CLAUDE.md` is stale. It still says the repo has no source files and implementation has not begun, which is now false. This will mislead future agent passes.
- `.codex/config.toml` sets `log_dir = "/codex_files/codex_log"`. That is an absolute root path, not the repo-local `codex_files/` directory. It is likely not what you intend.
- `pyproject.toml` is workable for smoke testing, but it uses `setuptools.backends.legacy:build` and discovers packages with `include = ["src*"]`. That means the installed import namespace is the very generic package name `src`, which is brittle and confusing for a growing RL codebase.
- `src/utils/seed.py` is not yet strong enough for reproducible RL experiments on GPU. It seeds Python/NumPy/Torch, but does not set cuDNN deterministic flags or document nondeterminism limits.
- `src/utils/paths.py` uses second-resolution timestamps only. Two runs launched within the same second can collide. The current uniqueness test only passes because it sleeps for one second.
- `src/utils/logging.py` returns early if a logger name already has handlers. Reusing a logger name later with a different `log_file` will silently skip the new file handler.
- `src/utils/io.py` hardcodes `weights_only=True` in `torch.load`. That is fine for a minimal tensor-only checkpoint, but it may become a problem once checkpoints include optimizer state, RNG state, config objects, or richer metadata.
- The test suite currently proves scaffold sanity, not research correctness. There are no checks yet for config validation, portfolio constraints, reward math, masking behavior, batch schemas, or environment transitions.
- `scripts/run_smoke_test.py` is referenced in the repo docs, but the script does not exist yet. That creates a mismatch between declared workflow and actual workflow.

Project-management feedback:
- Update the docs immediately after each phase so they remain operational instructions rather than historical artifacts.
- Before Phase 2 expands, define concrete acceptance criteria for config objects and batch schemas; otherwise the offline and online paths will drift early.
- Consider renaming the top-level import package away from `src` before more code lands. That migration is cheap now and annoying later.
- Decide whether `.claude/` and `.codex/` are repo-owned collaboration assets or local-only state. The current ignore/tracking policy is mixed.

Recommendation:
- Fix the stale docs, `log_dir`, run-directory collision risk, and logger behavior before building the config/data/environment layers. Those are small fixes now and become annoying sources of non-reproducible behavior later.
