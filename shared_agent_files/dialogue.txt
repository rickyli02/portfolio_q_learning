New bounded task: `Phase 15K scalar bundle file IO foundation`

Goal:
- add the narrowest useful file-based persistence layer for the approved `CTRLEvalScalarBundle`
- let the repo save and reload one coherent scalar evaluation result derived from a `CTRLEvalRecordSet`, without reopening plotting, backtesting, or broader artifact-management policy

Why this is next:
- `src/eval/` now has:
  - record sets
  - pure derivation back to summaries and aggregate
  - a typed scalar bundle over those derived outputs
- the next clean seam is persistence for that scalar bundle so later evaluation/report code can consume saved scalar bundles directly instead of reconstructing them from path-level artifacts every time

Scope:
- `src/eval/`
- `src/eval/__init__.py`
- directly related unit tests under `tests/unit/`
- do not modify `src/algos/ctrl.py`, `src/train/`, `scripts/`, plotting/backtest modules, or config wiring

Required deliverables:
- one small file-IO module for scalar bundles, for example `src/eval/bundle_io.py`
- one helper to save one or more `CTRLEvalScalarBundle` entries to a file path
- one helper to load those `CTRLEvalScalarBundle` entries back from a file path
- preserve the full approved `CTRLEvalScalarBundle` payload:
  - `seeds`
  - `summaries`
  - `aggregate`
- and within the embedded scalar objects preserve:
  - all `CTRLEvalSummary` fields
  - all `CTRLEvalAggregate` fields

Preferred narrow interpretation:
- use a simple explicit on-disk format such as JSONL with one scalar-bundle object per line
- because the bundle is already scalar-only, keep the on-disk format plain and human-readable
- return typed `CTRLEvalScalarBundle` objects on load
- fail fast with clear exceptions for:
  - nonexistent file
  - malformed JSON
  - valid JSON wrong top-level shape
  - missing required field
  - wrong scalar/list type
  - inconsistent bundle-level lengths:
    - `len(seeds) == len(summaries)`
    - `aggregate.n_episodes == len(summaries)`
- optional target-related fields should roundtrip as `None` when absent

Explicit testing requirements:
- tests must prove exact roundtrip behavior, not just plausibility
- at least one roundtrip test must compare every scalar-bundle field of a saved entry against the loaded entry:
  - exact seed list equality
  - exact per-summary field comparison for every summary
  - exact aggregate field comparison
- if multiple-bundle save/load is implemented, tests must prove order preservation exactly
- malformed-content tests should include:
  - nonexistent file
  - invalid JSON / malformed line
  - valid JSON wrong top-level shape
  - missing required field
  - wrong scalar/list type at bundle level
  - wrong scalar type inside embedded summary/aggregate fields
  - `len(seeds) != len(summaries)`
  - `aggregate.n_episodes != len(summaries)`
- include at least one live integration roundtrip from `bundle_from_record_set(...)`
- avoid weak tests that only check counts or that values are merely present

Unit tests should cover at least:
- save then load roundtrip for one bundle with exact seed, summary, and aggregate field comparison
- save then load roundtrip for multiple bundles preserving order
- `None` target-related fields roundtrip correctly
- nonexistent path on load is rejected cleanly
- malformed on-disk content is rejected cleanly
- wrong scalar/list types are rejected cleanly
- bundle-level consistency mismatches are rejected cleanly
- public API exports if added

Explicitly in scope:
- minimal typed save/load helpers for `CTRLEvalScalarBundle`
- exact disk roundtrip for scalar evaluation bundles
- public exports in `src/eval/__init__.py` if needed

Explicitly out of scope:
- plotting/report generation
- broader run-directory or artifact-management policy
- record / record-set / aggregate / summary logic changes beyond tiny local reuse if helpful
- compression or binary artifact conventions
- trainer integration
- stochastic evaluation

Stop conditions:
- stop if this naturally expands into a general report artifact layer rather than simple scalar-bundle persistence
- stop if the cleanest implementation would require broader changes outside `src/eval/`
- stop if plotting/report concerns become entangled with the IO helpers

Implementation note:
- keep the existing non-blocking assumptions unchanged:
  - record and bundle containers remain minimally mutable/frozen as already approved
  - bundle IO may rely on the current upstream contract that derived bundles are internally seed/summary-count consistent
