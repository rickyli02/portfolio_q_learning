## Ricky's safety-guard request: analysis of risky operations in current models

The following operations in src/models/ can fail silently or produce inf/nan:

1. `torch.exp(self.log_phi1)` [gaussian_actor.py line 101]:
   - Overflow risk if log_phi1 >> 0 (e.g., gradient runaway → φ₁ → inf).
   - Result: mean_action → inf, sample → inf, log_prob → nan.

2. `torch.exp(-self.log_phi2_inv)` [gaussian_actor.py line 106]:
   - If log_phi2_inv << 0 (large precision), φ₂ overflows to inf.
   - If log_phi2_inv >> 0 (small precision), φ₂ underflows to 0 → variance = 0 → log_prob → -inf.

3. `self.phi2 * torch.exp(self.phi3 * (self.horizon - t_t))` [gaussian_actor.py line 121]:
   - If phi3 > 0 and time_to_go is large: exponential overflow → variance → inf → sample noise → inf.
   - If phi3 < 0 and time_to_go is large: variance → 0 → log_prob → -inf.

4. `torch.log(var)` in entropy [gaussian_actor.py line 207]:
   - var = phi2 * exp(...) which is always > 0 by construction (log-param storage).
   - Safe under normal conditions, but produces -inf if var underflows to 0.0 (float32 denormal).

5. `torch.exp(-self.theta3 * time_to_go)` [quadratic_critic.py line 105]:
   - If theta3 < 0 and time_to_go large: exponential overflow → quad term → inf.

6. `torch.linalg.solve(cov, B)` [oracle_mv.py]:
   - Already raises torch.linalg.LinAlgError for singular/near-singular inputs.
   - No silent failure; but no user-friendly message wrapping it.

Suggested plan for safety guards (for future bounded task):
- Add value clamping or parameter projection at the algorithm/trainer layer
  rather than inside the model modules (keeps models clean and testable).
- Alternative: add a `validate_parameters()` method to ActorBase/CriticBase
  that checks for inf/nan parameters and raises informative errors.
- For the oracle, wrap the solve in a try/except that re-raises with env context
  (mu, sigma values) for easier debugging.
- Do NOT add guards inside forward() that silently correct parameter drift;
  this masks training bugs. Guards should assert or raise, not clamp and continue.

This is a planning note. No code changes proposed here; Codex or user should
assign a bounded task if guards are to be implemented.
