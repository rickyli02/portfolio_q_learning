Required deliverables:
- one minimal typed aggregate summary structure, for example a dataclass, containing at least:
  - number of episodes evaluated
  - mean terminal wealth
  - min terminal wealth
  - max terminal wealth
  - mean terminal gap when `target_return_z` is provided
  - count or rate of episodes meeting/exceeding target when `target_return_z` is provided
- one helper that runs deterministic evaluation across an explicit sequence of seeds and returns:
  - per-episode summaries, or a clear path to access them
  - the aggregate summary
- the helper should build on the already-approved `eval_summary(...)` rather than duplicating summary extraction logic

Preferred narrow interpretation:
- accept explicit runtime objects (`actor`, `env`, `w`, `seeds`, optional `target_return_z`)
- evaluate one episode per seed
- keep outputs scalar-first and inspection-friendly
- if `target_return_z` is omitted, make the aggregate target-gap fields explicitly `None` or similarly absent by contract

Explicit testing requirements:
- tests must prove exact aggregation behavior on known per-episode summaries, not only plausibility
- at least one test should compare aggregate fields against aggregates computed manually from individual `eval_summary(...)` calls over the same seed list
- if target-hit count/rate is included, tests must verify its exact value on a controlled seed list
- avoid tests that only assert inequalities like "mean is between min and max" unless they are secondary to stronger exact checks

Unit tests should cover at least:
- aggregate episode count matches the number of provided seeds
- aggregate fields exactly match manual aggregation from per-episode summaries
- target-gap aggregate behavior when `target_return_z` is provided
- behavior when `target_return_z` is omitted
- aggregate types are plain scalars / typed fields rather than tensors where practical
- public API exports if added

Explicitly in scope:
- typed multi-episode deterministic aggregate summary boundary
- one aggregate evaluation helper over explicit seeds
- narrow unit coverage for exact aggregation correctness

Explicitly out of scope:
- stochastic rollout evaluation
- plotting/report generation
- benchmark comparison tables
- config-dispatch wiring
- file IO for aggregates
- broader backtesting framework or portfolio analytics

Stop conditions:

- stop if the cleanest implementation would require broad changes outside `src/eval/`
- stop if plotting/report formatting becomes entangled with the aggregate helper
