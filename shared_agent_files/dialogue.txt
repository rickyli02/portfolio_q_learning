Process self-assessment: review stringency and test density

---

ON THE REVIEW PROCESS: LARGELY NECESSARY

Every Codex blocking finding across 20+ phases was substantively correct:

- Phase 17B: warn_if_ill_conditioned called before the try/except, so singular inputs
  emitted a UserWarning before raising ValueError — a real behavioral contract violation.
- Phase 17C: .double() mutates the module in place; both dtype branches were the same
  object. Oracle comparison was a pure self-comparison. Both genuine correctness bugs.
- Phase 17D: "extreme regime" at t=0.3, phi3=90 gives exp(63) which is finite in float32;
  the overflow claim was factually wrong.
- Phase 20A first pass: silently accepting policy_type="nonexistent" while constructing
  GaussianActor is a real design flaw for a module that documents a supported surface.
- Phase 20A second pass: adding a config field without updating the YAML documented as
  matching dataclass defaults is a real documentation consistency violation.
- Phase 20B: test_script_uses_seed_from_config asserted only rc==0 — would pass even if
  the script ignored the path and used hardcoded constants.

None of these were pedantry. Each blocked a silent correctness problem or misleading
guarantee. The review-to-approval ratio (often 2–3 rounds) reflects implementation
quality on the first pass, not excessive standards.

One area where process adds overhead without proportional benefit: the multi-round cycle
on phases that are mostly correct but have one narrow fixable issue. A lighter first-pass
checklist on simpler phases could reduce latency without lowering correctness standards.

---

ON TEST COUNT: MODERATELY HIGH, WITH DIMINISHING RETURNS IN PLACES

At ~900 tests for a repo still in scaffolding, density is high. Assessment by category:

High-value tests (catch non-obvious regressions):
- Policy-contract tests (Phase 19A): would fail if stochastic/deterministic roles swapped.
- Singular-path regression in oracle (Phase 17B): proves no warning on the failing path.
- Separate-instance dtype proof (Phase 17C): would fail if .double() mutation pattern returned.
- Unsupported-config rejections (Phase 20A): proves the supported surface is explicit.

Lower-value tests (formatting stability over correctness):
- Marker-presence tests ("assert 'ctrl_mean_tw' in out") check formatting, not behavior.
  They are not wrong but fragile to cosmetic changes.
- Frozen-dataclass immutability tests (pytest.raises on frozen dataclass assignment) test
  Python's own behavior, not repo logic.
- Several argument-count / return-code tests that pair a nonzero-rc assert with a separate
  text-content assert could reasonably be one combined test.

Estimate: test count could be 20–30% lower without losing meaningful coverage. The main
driver of over-testing is that each bounded phase adds a self-contained file, and the
cumulative overhead grows without periodic consolidation.

---

OVERALL: process is well-calibrated; test density is at the upper edge but defensible
given the research-code context where correctness guarantees matter more than velocity.
